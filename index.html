LayoutInflater.inflate

   前段时间提取拨号键盘控件，使用的是XML资源文件，结果一直没有内容，用Layout-inspector查看也是没有内容，几经排查发现原来是在引入ui时使用inflate（）不当导致的，具体看源码

/**
 * Inflate a new view hierarchy from the specified xml resource. Throws
 * {@link InflateException} if there is an error.
 * 
 * @param resource ID for an XML layout resource to load (e.g.,
 *        <code>R.layout.main_page</code>)
 * @param root Optional view to be the parent of the generated hierarchy.
 * @return The root View of the inflated hierarchy. If root was supplied,
 *         this is the root View; otherwise it is the root of the inflated
 *         XML file.
 */
 View convertView = inflater.inflate(R.layout.layout_dialpad,null);

因为我是直接继承LinearLayout，作为控件容器，但是父容器却置为null，这就导致控件和资源文件的脱节。

ubuntu下遇到识别不了的手机

   换了东家，换到ubuntu下的开发环境，各种折腾，装好开发环境AS，连上手机，为什么我的荣耀8还是不行呢，配好adb环境变量，adb device,提示手机未被授权，根据提示网址科学上网查看，原来坑爹的ubuntu需要多做一些设置：
如果您在 Ubuntu Linux 上开发，则需要为想要在开发中使用的每一种设备类型添加一个包含 USB 配置的 udev 规则文件。在规则文件中，每一个设备制造商都由一个唯一的供应商 ID（如 ATTR{idVendor} 属性所指定）标识。要查看供应商 ID 列表，请参见下文的 USB 供应商 ID。要在 Ubuntu Linux 上设置设备检测，请执行以下操作：

以 root 身份登录，并创建此文件：/etc/udev/rules.d/51-android.rules。

使用下面的格式将各个供应商添加到文件中：
SUBSYSTEM=="usb", ATTR{idVendor}=="0bb4", MODE="0666", GROUP="plugdev"

在本例中，供应商 ID 为 HTC 的 ID。MODE 赋值指定读/写权限，GROUP 则定义哪个 Unix 组拥有设备节点。

注：取决于您的环境，规则语法可能稍有不同。如有需要，请查阅适用于您的系统的 udev 文档。有关规则语法的概述，请参阅编写 udev 规则的指南。
现在，请执行：
chmod a+r /etc/udev/rules.d/51-android.rules

   后来又去网上搜，结合帖子Ubuntu/Mac彻底解决手机ADB识别问题中提到的
先用usb线连接手机
终端输入lsusb，获取所有usb信息，如下
Bus 002 Device 005: ID 12d1:107e Huawei Technologies Co., Ltd.
ID后面就是usb设备商编号，再结合之前的操作就可以了。

面试问题总结

  • 介绍你工作中遇到的问题及解决方法

   之前有个文件下载接口调试的时候，用浏览器下载完全没问题，可是我自己调用的时候却一直连接不上，后来就使用抓包工具抓取浏览器调用的包和app调用的包，对比之后发现，原来是因为浏览器会把非英文及数字的输入URL编码，而下载链接中有中文，app端没有对中文进行URL编码，所以浏览器可以调用，app调用同样的链接是行不通的。

   第二个案例是我在工作中遇到蓝牙外设开发，一开始使用service处理，但是在遇到app退到后台的之后再返回会出现socket连接已经断开，而socket端口还占用，导致重新连接不上，为了更好的用户体验，我们单独开了一个进程处理蓝牙通信服务，使用aidl的RemoteCallbackList实现方法回调，通知主进程蓝牙服务的变化。

  • Intent通信可以发送哪些类型的数据，如果要发送自定义的对象，必须实现哪个接口，为什么

   Intent只能发送基本类型的，如果要发送自定义对象，必须实现Parcelable或者Serializable，这二者的区别是Serializable使用反射实现序列化和反序列化，需要大量的I/O操作，而Parcelable自己实现了封送和解封送（marshalled &unmarshalled）不需要反射，而且数据也存在Native内存中，所以效率更高。Parcelable和Parcel的关系：Parcelable接口定义了封送和解封送的契约和规范，Parcel是一种最快速的序列化/反序列化机制，Parcelable的底层容器对象类是Parcel，是专为android进程间通信设计的。一定要注意，Intent的Bundler是在使用Binder机制进行数据传递的，能使用的Binder的缓冲区是有大小限制的（有些手机默认是2M），每个进程默认有16个Binder线程，所以每个线程能占用的缓冲区会更小。（大概有128k左右）所以看到“The Binder transaction failed because it was too large.”这类异常时你会知道怎么处理了。而且前面说了，Parcelable是使用Parcel容器实现封送和解封送的，去看Parcel会发现有些方法是Native即是用本地c代码实现的，是跟机子绑定的，所以最好不要用Parcelable进行持久化操作。

  • 线程间通信问题，UI线程和子线程的区别，可以直接在子线程中绑定UI线程，实现更新数据吗，为什么

   照例是handler、looper、message、messagequeue那一套，涉及到JVM底层，这样设计是有它的好处的，因为如果直接更新的话（不考虑内存泄漏问题），也是可以实现的，但是会出现什么情况呢，没有线程去刷新UI界面，因为实际上每个线程中的数据在JVM的堆栈中在外部都有自己的缓存，所以光更新ui进程是有可能出现数据缓存问题的，就是你更新了数据，但是显示的还是旧的数据，而且既然ui线程被用来做这个，那么就没有线程来刷新ui了，其他网上的解释参见Android子线程真的不能更新UI么,有时间在了解一下JVM原理。

线程间消息机制简析

   每个app启动的时候，都会创建自己的进程，并且生成一个处理四大组件的主线程，主线程管理一个消息队列，为避免ANR，我们必须把耗时操作放在子线程中。子线程和主线程的交互通讯就要通过handler、looper、message、messagequeue机制处理。我们先看看平常用的handler构造函数，注释中写明，默认构造函数关联的是当前的线程的Looper。

/**
 * Default constructor associates this handler with the {@link Looper} for the
 * current thread.
 *
 * If this thread does not have a looper, this handler won't be able to receive messages
 * so an exception is thrown.
 */
public Handler() {
    this(null, false);
}

   在Activity中有activityThread，默认创建looper。在主线程中构造handler可以直接使用，而在子线程中如果没有先调用Looper.preper()直接创建handler会报异常。

mLooper = Looper.myLooper();
if (mLooper == null) {
        throw new RuntimeException(
            "Can't create handler inside thread that has not called Looper.prepare()");
}

   Loop是循环的意思，实际上Looper也是维持一个消息队列的死循环，当没有消息时这个循环是阻塞的，直到handler发送消息到队列中，阻塞消除，处理消息队列，并下发消息，处理完消息，重新回到阻塞状态。

 /**
 * Run the message queue in this thread. Be sure to call
 * {@link #quit()} to end the loop.
 */
public static void loop() {
    final Looper me = myLooper();
    if (me == null) {
        throw new RuntimeException("No Looper; Looper.prepare() wasn't called on this thread.");
    }
    final MessageQueue queue = me.mQueue;

    // Make sure the identity of this thread is that of the local process,
    // and keep track of what that identity token actually is.
    Binder.clearCallingIdentity();
    final long ident = Binder.clearCallingIdentity();

    for (;;) {//1
        Message msg = queue.next(); // might block
        if (msg == null) {
            // No message indicates that the message queue is quitting.
            return;
        }

        // This must be in a local variable, in case a UI event sets the logger
        final Printer logging = me.mLogging;
        if (logging != null) {
            logging.println(">>>>> Dispatching to " + msg.target + " " +
                    msg.callback + ": " + msg.what);
        }

        final long traceTag = me.mTraceTag;
        if (traceTag != 0) {
            Trace.traceBegin(traceTag, msg.target.getTraceName(msg));
        }
        try {
            msg.target.dispatchMessage(msg);//2
        } finally {
            if (traceTag != 0) {
                Trace.traceEnd(traceTag);
            }
        }

        if (logging != null) {
            logging.println("<<<<< Finished to " + msg.target + " " + msg.callback);
        }

        // Make sure that during the course of dispatching the
        // identity of the thread wasn't corrupted.
        final long newIdent = Binder.clearCallingIdentity();
        if (ident != newIdent) {
            Log.wtf(TAG, "Thread identity changed from 0x"
                    + Long.toHexString(ident) + " to 0x"
                    + Long.toHexString(newIdent) + " while dispatching to "
                    + msg.target.getClass().getName() + " "
                    + msg.callback + " what=" + msg.what);
        }

        msg.recycleUnchecked();
    }
}

   我们看源码，非主线程要和自己的Looper关联，必须调用prepar()，而且每个线程只能关联一个Looper，那不同的线程是如何获取自己的Looper，这就必须看ThreadLocal。

/** Initialize the current thread as a looper.
  * This gives you a chance to create handlers that then reference
  * this looper, before actually starting the loop. Be sure to call
  * {@link #loop()} after calling this method, and end it by calling
  * {@link #quit()}.
  */
public static void prepare() {
    prepare(true);
}
private static void prepare(boolean quitAllowed) {
    if (sThreadLocal.get() != null) {
        throw new RuntimeException("Only one Looper may be created per thread");
    }
    sThreadLocal.set(new Looper(quitAllowed));
}

   ThreadLocal中主要看两个方法set和get方法，简单的说，ThreadLocal保存的key是线程，value是Looper。详细的介绍见
Android的消息机制之ThreadLocal的工作原理

/**
 * Sets the current thread's copy of this thread-local variable
 * to the specified value.  Most subclasses will have no need to
 * override this method, relying solely on the {@link #initialValue}
 * method to set the values of thread-locals.
 *
 * @param value the value to be stored in the current thread's copy of
 *        this thread-local.
 */
public void set(T value) {
    Thread t = Thread.currentThread();
    ThreadLocalMap map = getMap(t);
    if (map != null)
        map.set(this, value);
    else
        createMap(t, value);
}
/**
 * Returns the value in the current thread's copy of this
 * thread-local variable.  If the variable has no value for the
 * current thread, it is first initialized to the value returned
 * by an invocation of the {@link #initialValue} method.
 *
 * @return the current thread's value of this thread-local
 */
public T get() {
    Thread t = Thread.currentThread();
    ThreadLocalMap map = getMap(t);
    if (map != null) {
        ThreadLocalMap.Entry e = map.getEntry(this);
        if (e != null)
            return (T)e.value;
    }
    return setInitialValue();
}
